# Quality: What Changes, What Stays

## 1. Quality Philosophy Shift

### 1.1 Old Model: Quality Is a Final Gate

In traditional development, quality assurance happens at the end. Code is written, then tested, then bugs are found and fixed. QA is a checkpoint between development and deployment.

### 1.2 New Model: Quality Is Embedded in Spec Writing

In the 24K Framework, quality starts with the spec. A bad acceptance criterion is a quality failure **before any code exists**. If the spec is incomplete, AI will generate code that satisfies the letter of the spec while missing its intent.

Quality gates exist at every stage:
- **Spec quality** → Spec DoD (Level 4 gate)
- **Code quality** → Human review of AI output
- **Test quality** → QA validation of test coverage
- **Ship quality** → Production monitoring against non-functional requirements

### 1.3 Quality Ownership

Everyone owns quality. AI amplifies quality practices — it does not replace human judgment about what "good" means.

## 2. Spec Quality: The First Gate

The most impactful quality intervention happens before any code is written. A clear spec produces clear code. A vague spec produces plausible-looking code that breaks in production.

### 2.1 Acceptance Criteria Quality Standards

Good acceptance criteria are:

- **Behavioral**: Describe what happens, not how it's built
- **Testable**: Can be automated into a test
- **Complete**: No implicit assumptions left unstated
- **Independent**: Each criterion can be verified independently

**Format**: BDD (Given/When/Then)

```
Given a user has just signed up
When they log in for the first time
Then they see the onboarding flow, not the empty dashboard
```

**Anti-patterns**:
- "The page should look good" → not testable
- "Handle errors properly" → not specific
- "Work like the existing feature" → implicit knowledge, not written down

### 2.2 The Spec DoD as Quality Gate

See [Specs](03_specs.md) section 2.5 for the complete Spec DoD checklist. The Scrum Master enforces this gate — no ticket enters Sprint Planning without passing the Spec DoD.

## 3. Code Quality in AI-Generated Codebases

### 3.1 What Degrades Faster with AI

AI generates code that is locally correct but globally inconsistent. Watch for:

- **Naming inconsistency**: AI uses different naming conventions in different files
- **Architecture drift**: AI creates new patterns instead of following existing ones
- **Over-engineering**: AI adds unnecessary abstractions, error handling for impossible cases
- **Copy-paste patterns**: AI duplicates code instead of reusing existing utilities

### 3.2 What Improves with AI

- **Test coverage**: AI generates tests reliably when given clear acceptance criteria
- **Documentation**: AI generates accurate inline documentation
- **Boilerplate correctness**: AI produces consistent boilerplate without typos or missing steps
- **Code formatting**: AI follows formatting rules consistently

### 3.3 The CONVENTIONS.md Anchor

Every project should have a `CONVENTIONS.md` file that is loaded into AI context. This file defines:

- Naming conventions (files, functions, variables, components)
- File structure patterns
- Architecture patterns to follow
- Anti-patterns to avoid
- Error handling conventions
- Test naming and structure conventions

The Technical Lead maintains this file. It is the mechanism by which human architectural decisions flow into AI-generated code.

## 4. Testing Strategy

### 4.1 AI-Generated Tests

AI generates tests reliably for:
- Happy path scenarios from acceptance criteria
- Standard input/output validation
- API contract testing
- Component rendering tests

**Human review required**: Are the AI-generated tests testing behavior (what the user experiences) or implementation details (how the code works internally)? Tests that test implementation details break on refactoring without catching real bugs.

### 4.2 Human-Written Tests

Humans write tests for:
- Edge cases not in the spec (discovered during development or review)
- Security boundary tests
- Performance-sensitive paths
- Complex business logic where intent matters more than structure

### 4.3 QA Validation

QA validates that the complete test suite — AI-generated and human-written — covers the spec's acceptance criteria. The question is not "do the tests pass?" but **"do the tests test the right things?"**

### 4.4 Test Levels

| Level | Generated By | Verified By | Tests What |
|---|---|---|---|
| Unit tests | AI (per function/component) | Developer | Individual component correctness |
| Integration tests | AI + human review | Developer | Service boundary interactions |
| E2E tests | Human-designed, AI-assisted | QA | Full user flow against acceptance criteria |
| Exploratory testing | Human only | QA | What the spec didn't think of |

## 5. The QA Role Transformed

### 5.1 QA as Spec Co-Author

QA writes acceptance criteria as part of the spec — not test cases after development. This shifts QA's contribution earlier in the pipeline where it has the most impact.

### 5.2 QA as User Advocate

QA validates that the **Why** was achieved, not just that the **What** was built. The question is: "Does this feature actually solve the user's problem?" — which requires human judgment, not automated tests.

### 5.3 QA as AI Test Curator

AI generates test suites from acceptance criteria. QA reviews and enhances them:
- Are edge cases covered?
- Do tests reflect user behavior or implementation details?
- Is the test data realistic?
- Are negative paths tested?

### 5.4 Exploratory Testing

The irreplaceable human contribution to quality. Exploratory testing cannot be scripted or AI-generated because its value comes from human curiosity, intuition, and domain knowledge.

Exploratory testing should focus on:
- What happens when the user does something unexpected?
- What happens at system boundaries (very large inputs, concurrent users, network failures)?
- What assumptions did the spec make that might be wrong?

## 6. Quality Metrics

| Metric | What It Measures | Healthy Range |
|---|---|---|
| Spec quality score | % of tickets at Level 4 before sprint entry | > 90% |
| AI generation success rate | % of AI outputs accepted without major revision | 60–80% |
| Acceptance criteria pass rate | First-pass QA validation against acceptance criteria | > 85% |
| Escaped defect rate | Bugs found in production vs. found in QA | < 10% |
| Mean time to spec invalidation | How quickly bad specs are caught | < 2 days |
| Test coverage of acceptance criteria | % of acceptance criteria with automated tests | > 95% |

**Key insight**: If the AI generation success rate is below 60%, the problem is almost certainly **spec quality**, not AI quality. Improve specs, not prompts.
